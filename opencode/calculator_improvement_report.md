# 복리 계산기(Calculator) 개선 보고서

## 🚀 현황 분석

현재 복리 계산기는 투자 금액, 기간, 수익률을 입력받아 자산 성장 추이를 시각화하는 핵심 기능을 제공합니다. 하지만 규모가 큰 데이터(긴 투자 기간) 처리 시 성능 저하 우려가 있으며, 코드 중복 및 유지보수성이 낮은 구조를 가지고 있습니다.

## 🛠 개선 필요 사항

### 1. 계산 알고리즘 최적화 (Performance)

- **문제점**: 현재 `calculateResult`는 시간(hour) 단위로 루프를 수행하여, 투자 기간이 길어질수록(예: 50년) 연산 횟수가 급격히 증가합니다 ($50 \times 8760 \approx 438,000$회). 이는 브록(Block) 현상을 유발할 수 있습니다.
- **해결책**: 복리 공식($A = P(1 + r)^n$)을 활용하여 **연(Year) 단위로만 루프를 수행**하도록 로직을 개편하여 연산량을 8,760배 감소시킵니다.

### 2. 코드 중복 제거 및 모듈화 (Maintainability)

- **문제점**: `page.tsx`와 `ResultChart.tsx` 두 곳에 동일한 **금액 변환 로직(`formatKoreanCurrencyFull`, `formatValueFull`)**이 중복 정의되어 있습니다.
- **해해책**: `lib/utils/format.ts`와 같은 공용 유틸리티 파일로 로직을 추출하여 단일 진실 공급원(Single Source of Truth)을 확보합니다.

### 3. UI/UX 정교화 (User Experience)

- **문제점**:
  - 입력값이 변경될 때 차트 애니메이션이 끊기거나 부자연스러울 수 있습니다.
  - `ResultChart`의 툴팁과 페이지의 `Callout` 디자인이 파편화되어 있습니다.
- **해결책**:
  - `framer-motion`을 활용하여 데이터 변경 시 차트의 Area가 부드럽게 전환되도록 개선합니다.
  - 프로젝트의 글로벌 디자인 시스템(Radix + Tailwind)에 맞춰 차트 컴포넌트의 스타일을 재정의합니다.

### 4. 데이터 안정성 (Data Integrity)

- **문제점**: `useStockSearch`와 마찬가지로, 계산기에서도 외부 입력(Slider, NumericInput)에 따른 예외 상황(예: 음수 수익률, 0원 투자 등)에 대한 에러 경계 처리가 필요합니다.
- **해해책**: 입력값 검증(Validation) 로직을 강화하고, 유효하지 않은 입력 시 `Callout`을 통해 즉각적인 피드백을 제공합니다.

## 📊 기대 효과

- **성능**: 대규모 연산 부하 제거로 저사양 기기에서도 끊김 없는 인터랙션 보장.
- **유지보수**: 코드 중복 제거를 통한 기능 수정 시 버그 발생 가능성 최소화.
- **완성도**: 디자인 일관성 확보를 통한 "Lovable"한 제품 경험 완성.
